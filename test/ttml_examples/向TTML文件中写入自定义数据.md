# 向TTML文件中写入自定义数据

### 📝 `metadata`：注入丰富元数据
推荐将需要由渲染引擎解析的、非可视化的数据放在 `<metadata>` 中，它是标准的元数据容器，能被解析器安全处理。

*   **`ttm:desc` - 段落/元素描述**：为任何元素提供人类可读的描述。例如，`<ttm:desc>This line is a chorus, the animation is 'fastPulse'</ttm:desc>` 可以描述该行的性质和动画效果。
*   **`ttm:title` - 文档标题**：在文档头部提供，主要用于识别内容。
*   **`ttm:copyright` - 版权信息**：声明文档的版权归属，有助于保护知识产权。
*   **`ttm:agent` - 标注贡献者**：声明歌词的演唱者（创作者），支持对唱场景下的多人区分。
*   **`EBU-TT-D` 元数据**：如果面向欧洲广播市场，可以使用`<ebuttm:documentMetadata>`等EBU特定的元数据命名空间。

### 🎨 `<styling>` / `<layout>` / `<region>`：定义视觉属性
你完全可以将视觉效果相关的数据放在样式或布局块中，它们是标准的一部分，专用于定义视觉属性。

*   **`<styling>`元素**：可用于定义标准样式（`tts:*`），也可以在自定义命名空间下定义私有动画集。例如，用 `<style xml:id="myFadeIn" myanim:type="fadeIn" myanim:duration="0.5s"/>` 来定义特定动画集。
*   **`<region>`元素**：可通过 `tts:origin`、`tts:extent` 等标准属性定义显示区域的位置和大小。

### 🧩 高级与平台特定扩展
这部分为你提供了更大的灵活性，可以根据具体需求进行定制，并能确保与平台良好集成。

*   **新建元素**：如果需要表达复杂结构（如为同一行定义多个独立动画轨），**新建一个自定义元素**是最好的选择。例如，`<myanim:track id="glow" target="#word1" type="pulse"/>`。但务必先声明对应的命名空间（如 `xmlns:myanim="..."`）。
*   **iTunes 扩展**：为 Apple Music 或 iTunes 制作歌词时，可使用 Apple 的 iTT 扩展。例如，标记段落结构：`<div itunes:song-part="Chorus">`。所有扩展都通过 Apple 特有的命名空间实现。
*   **`<ttm:item>` - 精细数据描述**：如需更精确地为某个元素附加键值对数据，`<ttm:item>` 是理想选择。它为精细的数据描述而生，可用于键值对的附加。
*   **`EBU-TT-D` 扩展**：针对欧洲广播特定场景，可以使用 `ebu:` 下的专有元数据命名空间。

### ⏱️ `<set>`：嵌入时间线关键帧
`<set>` 元素是 TTML2 中用于在特定时间点动态改变元素样式或属性的标准方式。它本质上允许你定义一个关键帧快照，这非常适合与你的自定义动画系统对接。

例如，你可以这样设置关键帧：
```xml
<p>
  <set begin="00:05.000" end="00:05.500" myanim:scale="1.0"/>
  <set begin="00:05.500" end="00:06.000" myanim:scale="1.5"/>
  <set begin="00:06.000" end="00:06.500" myanim:scale="1.0"/>
  A word with bounce animation.
</p>
```
但需要注意，这部分功能属于 **TTML 2.0** 标准。如果你的应用场景对稳定兼容性要求极高，那么提前引入 2.0 特性并非最佳选择。

### ⚠️ 不同配置文件下的选择与兼容性
不同配置文件的允许范围差异很大，需根据以下场景选择：

*   **通用/自研引擎（选择自由度高，限制少）**：
    *   如需交付最小风险、最通用的标准文件：优先使用 `metadata` 中的 `ttm:*` 元素。
    *   如需保留丰富的自定义数据，且无平台互操作要求：可混合使用 `metadata`、`styling`、自定义元素和属性。

*   **Apple Music / iTunes (iTT)（无样式定义能力，有特定扩展）**：
    *   可以在 `metadata` 中使用 `ttm:agent`、`ttm:name` 和 `ttm:title`，并能使用 `itunes:song-part` 扩展。
    *   **⚠️ 关键限制**：Apple 的渲染引擎会**忽略** `<head>` 部分中所有样式和布局的定义，但你的自定义引擎依然可以保留这些数据并在导出时删除。

*   **Netflix / BBC (IMSC)（精简、互操性强）**：
    *   核心在于 `metadata`、`styling` 和 `layout`，同样**严禁在内容元素上使用内联样式属性**。
    *   理论上可以使用自定义命名空间，但一般不建议，以免破坏严格的互操作性约束。

*   **欧洲广播电视 (EBU-TT-D)（保留官方扩展能力）**：
    *   `metadata` 容器支持，但应使用 `ebu:` 命名空间下的元数据以取代自定义数据。

### 💡 总结与最佳实践
总的来说，如果数据属于文档级信息，优先放入 `<metadata>`；如果是样式或布局相关，应放在 `<styling>` 或 `<layout>` 里。
1.  **`<metadata>`**：适用于文档级元数据（如标题、描述、演唱者）。若数据是全文档范围的，建议优先放在此处。
2.  **`<styling>` / `<layout>`**：适用于可复用的样式和布局属性，数据属于样式或布局层面。
3.  **新建元素**：如果需要表达复杂、多值的结构化数据。
4.  **自定义属性**：适合向现有元素附加简单值（如 `myanim:duration="0.3s"`），简单快捷。
